# 水之波光 · 触境
## The Water's Gleam · Tactile Realm
### 设计推导法 · 推导引擎 v10.0.0
**性质**：方法论。连接哲学与实例的推导协议。不输出设计值，只输出推导流程与裁决规则。
---
## 3.0 推导引擎概述
推导引擎是将**目的**转化为**形态**的因果机器。它本身不产生美，它只确保美的产生过程是诚实的。
**输入**：经过验证的目的（任务分析档案）  
**处理**：约束激活 → 权重分配 → 可行空间挤压 → 冲突裁决 → 来源标注  
**输出**：帕累托最优解集 + 具体选择记录 + 完整推导链
**核心纪律**：
- 无目的输入，推导禁止启动。
- 无来源标注，输出视为无效。
- 触及钻石边界，任何目的不可豁免。
---
## 3.1 目的验证协议
目的不是推导的装饰品，是推导的启动器。**目的无效，约束链无法激活。**
### 3.1.1 目的陈述格式
有效目的必须可被还原为**具体用户行为序列**。格式：
```
目的：[用户角色] 在 [场景] 中，需要完成 [任务]，以达成 [价值]。
最小信息集：[完成任务所需的最少界面元素]
高频路径：[用户最频繁执行的行为序列]
零态场景：[内容为空/加载失败/无权限时的出现频率与上下文]
```
**示例（有效）**：
> 目的：开发者在代码编辑器中，需要快速定位语法错误并修复，以维持心流。
> 最小信息集：行号、错误标记、错误消息、快速修复入口。
> 高频路径：看到红线 → 悬停查看消息 → 点击修复 → 继续编码。
> 零态场景：无错误时错误面板为空（低频，非核心任务阻塞）。
**示例（无效）**：
> 目的：提升用户粘性。
> 诊断：无法还原为行为序列。"粘性"是指标，不是任务。
### 3.1.2 目的验证三问
| 问 | 验证标准 | 未通过的后果 |
|---|---|---|
| **任务还原审** | 能否写出5步以内的用户行为序列？ | 目的过于抽象，约束链无法启动。 |
| **最小信息审** | 能否列出完成任务所需的最少元素？超出部分是否为负债？ | 界面臃肿，信息噪音增加认知负荷。 |
| **需求对齐审** | 这个目的是否来自用户研究/任务分析，而非内部假设？ | 设计服务于假设中的人，而非真实的人。 |
### 3.1.3 目的权重分配
目的验证通过后，进入权重分配。**目的不改变钻石边界，目的改变钻石与其他约束的相对权重。**
**权重分配表模板**：
| 约束类别 | 本项目权重 | 理由 |
|---|---|---|
| 韦伯阈值（P-001） | 极高/高/中/低 | 文本密集型任务需丰富层级，权重极高；沉浸式视频需界面退后，权重降低 |
| 空间频率（P-002） | ... | ... |
| 工作记忆（P-020） | ... | ... |
| 能耗预算（R-003） | ... | ... |
| ... | ... | ... |
**规则**：
- 钻石约束不可标为"忽略"，只能标为"权重降低"。
- 权重分配必须有目的层面的理由，不能凭直觉。

## 3.2 约束激活与可行空间挤压
### 3.2.1 约束激活流程
```
目的验证通过
    ↓
激活所有钻石约束（全部激活，权重由目的分配）
    ↓
激活所有钢铁约束（全部激活，权重由目的分配）
    ↓
激活相关内容逻辑约束（由最小信息集推导）
    ↓
引入工程约定与项目基因（陶土层，显式声明）
    ↓
挤压可行空间
```
### 3.2.2 可行空间挤压规则
**硬边界排除**：
- 任何方案若违反钻石约束，直接排除，无需讨论。
- 任何方案若违反钢铁约束，在当前技术代际下排除。
- 内容逻辑约束违反（如用颜色作为唯一层级区分）直接排除。
**软边界协商**：
- 陶土层约束（工程约定、项目基因）可在项目上下文内调整。
- 调整必须记录理由，并标注 `[ENG: 调整为X，原因: Y]`。
### 3.2.3 可行空间记录格式
```
可行空间挤压记录
├── 硬边界排除方案
│   ├── 方案A：纯黑背景 + 高频零态 → 违反 P-001（韦伯阈值），排除
│   ├── 方案B：1px 红色分割线 → 违反 D-003（Pentile 彩边），排除
│   └── ...
├── 剩余可行空间
│   ├── 方向1：#121212 背景 + 完整视觉层级
│   ├── 方向2：#F8F9FA 背景 + 高对比层级
│   └── ...
└── 帕累托最优解集：[方向1, 方向2, ...]
```

## 3.3 帕累托最优解集与具体选择
### 3.3.1 帕累托最优的定义
在可行空间内，**无法在不冒犯至少一个约束的情况下进一步优化**的解，构成帕累托最优解集。
**关键理解**：
- 帕累托最优解通常不止一个。
- 多个帕累托最优解之间没有绝对的"更好"，只有"更适合当前项目上下文"。
- 选择哪一个，由**项目基因 + 工程约定 + 团队共识**决定。
### 3.3.2 具体选择记录
选择帕累托最优解集中的某一个时，必须记录：
```
选择：[具体值]
来源标注：
  - [PHYS] 支撑该选择的物理约束
  - [ENG] 支撑该选择的工程约定
  - [GENE] 支撑该选择的项目基因
  - [PURPOSE] 目的权重分配结果
未选择其他帕累托解的理由：
  - [解B] 未被选择，因为 [原因]
负债标记（如有）：
  - TODO: [DEBT] [原因] [演进路径]
```
**示例**：
```
选择：背景色 #121212
来源标注：
  - [PHYS: D-004] OLED 子像素关闭特性降低能耗
  - [PHYS: P-001] 保留足够亮度空间供层级表达（差异 > 2% 可感知）
  - [PURPOSE] 项目为文本密集型任务，零态高频出现，需完整视觉层级
  - [GENE] Helix 品牌偏好暗色（已验证不违反钻石约束）
未选择 #000000 的理由：
  - 纯黑在零态高频场景下违反 P-001，视觉层级压缩不可接受
未选择 #F8F9FA 的理由：
  - 项目基因偏好暗色，且用户主要在暗环境使用
负债标记：无
```

## 3.4 冲突裁决引擎
### 3.4.1 冲突类型
| 类型 | 定义 | 示例 |
|---|---|---|
| **钻石-钻石冲突** | 两个钻石约束在同一目的下无法同时满足 | 极窄视口（320px）下，字号层级差 ≥ 20% 与标题不换行超过3行 |
| **钻石-钢铁冲突** | 钻石约束与钢铁约束冲突 | OLED 纯黑节能（钢铁） vs 韦伯阈值层级表达（钻石） |
| **钻石-陶土冲突** | 钻石约束与工程约定/项目基因冲突 | 品牌色水蓝在特定背景下对比度不足 |
| **目的-约束冲突** | 目的要求与约束边界冲突 | 目的要求"极简无层级界面"，但内容本身有三级层级 |
### 3.4.2 冲突裁决顺位
```
第一优先：生存权（钻石约束不可妥协）
    ├── 视觉边界：对比度 ≥ 4.5:1，无 >3Hz 闪烁，热区 ≥ 44px
    ├── 认知边界：工作记忆不超载，预测误差可控
    └── 可访问性：WCAG AA，色觉障碍双重编码
        ↓
第二优先：识读权（内容必须可被获取）
    ├── 内容唯一：这是什么？我在哪？我能做什么？
    ├── 层级有序：视觉层级映射信息层级
    └── 排版可读：行宽、行高、字号在生理舒适区
        ↓
第三优先：反馈权（因果链必须完整）
    ├── 因果不虚：每个行为有可感知反馈
    ├── 涟漪完整：升起-扩散-消散-复原
    └── 状态互斥：逻辑层裁决，CSS 消费确定状态
        ↓
第四优先：流转权（质感与灵动）
    ├── 微光流转：因动而动，零闲置能耗
    ├── 动效细节：缓动、时长在感知窗口内
    └── 视觉质感：深度暗示、遮罩、阴影
```
### 3.4.3 冲突裁决流程
```
1. 识别冲突双方（约束A vs 约束B）
2 判定双方硬度（钻石/钢铁/陶土/目的）
3. 查找裁决顺位
    ├── 若一方为钻石，另一方为陶土 → 钻石胜出，陶土让步
    ├── 若双方同为钻石 → 进入同级钻石裁决
    └── 若涉及目的权重 → 验证目的权重分配是否合理
4. 执行裁决，记录让步方
5. 标记负债：TODO: [DEBT] + 原因 + 演进路径
6. 触底检查：是否触及生存权？是 → 停止推导，修正目的或技术方案
```
### 3.4.4 同级钻石裁决示例
**冲突**：320px 视口下，字号层级差 ≥ 20%（P-001） vs 标题不换行超过3行（阅读节奏，P-004/P-005 推论）。
**裁决**：
- 双方均为钻石约束的推论。
- 目的权重：文本阅读任务下，阅读节奏（心流维持）权重略高于字号层级差的绝对精确。
- **裁决**：字号降至 26px（层级差约16%，低于理想阈值），标记负债。
- **负债**：`TODO: [DEBT] 320px 视口下标题层级差不足20% | 原因: 视口宽度硬约束 | 演进: 视口 > 375px 时恢复标准层级差`

## 3.5 降级与重协商协议
### 3.5.1 降级不是妥协，是约束链的诚实重协商
当渲染环境、设备性能、用户偏好发生变化时，约束链必须动态重协商。**降级是"在不同约束集下重新推导"，不是"减少功能"。**
### 3.5.2 重协商时机
| 时机 | 触发条件 | 响应方式 |
|---|---|---|
| **设计时重协商** | 目标环境在构建前已确定 | 静态选择，直接输出适配代码 |
| **运行时重协商** | 环境在用户设备上动态变化 | 媒体查询、特性检测、性能监控 |
### 3.5.3 降级阶梯（从流转权开始）
| 阶梯 | 内容 | 触发条件 |
|---|---|---|
| **第一降级** | 引导流转停止 | `prefers-reduced-motion`、低电量、低端设备 |
| **第二降级** | 微光流转视觉停止（纯黑模式下自动触发） | 背景为纯黑且视觉幅度低于韦伯阈值 |
| **第三降级** | 动效细节降级（缓动→ease-out，时长缩短） | FPS < 30、低端 GPU |
| **第四降级** | 视觉质感降级（遮罩→纯色，半透明→实色） | 极端性能受限、高对比度模式 |
### 3.5.4 不可降级的底线
| 底线 | 原因 |
|---|---|
| 反馈必须存在 | 因果不虚（公理二） |
| 对比度 ≥ 4.5:1 | 钻石约束 P-001 |
| 热区 ≥ 44px | 钻石约束 P-010 |
| 状态逻辑互斥（JS环境） | 钢铁约束 R-001 |
| 零态自证 | 钻石约束 P-014（闭合性）+ P-003（情绪时间） |
| 设计令牌零硬编码 | 工程底线 |

## 3.6 来源标注协议
### 3.6.1 标注义务
**无标注 = 推导无效。** 每一个输出的设计值必须携带完整的来源标注。
### 3.6.2 标注格式
```
[值名]: [具体值]
├── [PHYS] [约束ID] [约束名称] → [该约束如何支撑此值]
├── [ENG] [约定名称] → [该约定如何简化此值]
├── [GENE] [基因项] → [品牌/项目身份如何影响此值]
└── [PURPOSE] [目的权重项] → [目的如何分配权重导致此值]
```
### 3.6.3 标注示例
```
背景色: #121212
├── [PHYS: D-004] OLED 子像素关闭 → 降低能耗
├── [PHYS: P-001] 韦伯阈值 → 保留 >2% 亮度差异空间供层级表达
├── [PURPOSE] 文本密集型任务 → 零态高频，需完整视觉层级
└── [GENE] Helix 暗色偏好 → 品牌身份表达（已验证不违反钻石约束）
字号: 16px
├── [PHYS] 标准阅读距离人眼分辨率 → 最小舒适正文字号
├── [ENG] 浏览器默认基准 → 跨平台一致性
└── [PURPOSE] 通用文本阅读 → 信息密度与可读性平衡
圆角: 8px
├── [ENG] 4px 网格倍数 → 简化协作与心算
├── [PHYS: P-002] 空间频率 → 配合抗锯齿降低高频噪声
└── [PURPOSE] 移动端卡片界面 → 软化边界，促进闭合性
```
### 3.6.4 常见标注错误
| 错误 | 示例 | 修正 |
|---|---|---|
| 陶土冒充钻石 | `[PHYS] 圆角 8px 是格式塔必然` | `[ENG] 嵌套比例 1/2 是工程约定；[PHYS] 连续性促进闭合性` |
| 基因冒充约束 | `[PHYS] 涟漪层必须是水蓝` | `[PHYS] 涟漪层必须单色相；[GENE] 水蓝是 Helix 品牌色` |
| 目的缺失 | `[PHYS] 暗色模式降低眩光` | 补充 `[PURPOSE] 用户主要在暗环境使用` |
| 标注不完整 | `背景色: #121212 [PHYS]` | 必须列出所有支撑来源 |

## 3.7 审计清单：十一审执行协议
审计不是一次性检查，是**每一步推导的强制校验**。
| 审 | 执行时机 | 审计问题 | 未通过的处理 |
|---|---|---|---|
| **目的适配审** | 推导启动前 | 目的是否通过三问验证？是否与真实需求对齐？ | 停止推导，修正目的。 |
| **来源标注审** | 每个设计值输出时 | 是否标注了 [PHYS]/[ENG]/[GENE]/[PURPOSE]？陶土是否冒充了钻石？ | 输出无效，重新标注。 |
| **因果审** | 组件设计时 | 每个元素为何存在？删除它是否减少价值？ | 删除无因元素。 |
| **层级审** | 信息架构时 | 视觉层级是否映射信息层级？是否用颜色作为唯一区分？ | 重构层级表达。 |
| **生命审** | 交互设计时 | 涟漪四阶段是否完整？状态互斥是否由逻辑层裁决？ | 补全断链，迁移互斥逻辑。 |
| **结构审** | 元素审查时 | 删除后，可读性/聚焦/边界是否下降？ | 三项均无影响 → 删除。 |
| **生理审** | 可访问性审查时 | 是否违反对比度/热区/闪烁/色觉障碍包容？ | 触及生存权，必须修正。 |
| **触境审** | 交互原型测试时 | 闭上眼睛，只凭指尖——能感受到回应吗？ | 补全触觉/听觉反馈通道。 |
| **容器审** | 内容填充测试时 | 最长文本替换后，容器是否仍有呼吸空间？ | 增加冗余。 |
| **能耗审** | 性能审查时 | 闲置时是否仍在消耗非必要资源？交互流转是否仅在用户行为时触发？ | 停止闲置动画。 |
| **约束链审** | 最终交付前 | 是否假设了不存在的技术能力？是否遗漏了约束环节？约束陈述是否精确到可量化阈值或明确标注 [ENG]？ | 补全约束链，修正模糊陈述。 |

## 3.8 推导引擎的元纪律
### 3.8.1 引擎不自指
推导引擎本身也受约束链约束：
- 引擎的规则不能违反卷一的公理。
- 引擎的流程不能假设设计者拥有无限认知资源。
- 引擎的输出不能预设特定审美。
### 3.8.2 引擎可演进
当认知科学或显示技术出现新边界时：
1. 新约束进入卷二约束库。
2. 引擎流程在卷三更新。
3. 旧约束被淘汰时，从全文档彻底移除。
### 3.8.3 引擎不替代判断
帕累托最优解集中的最终选择，需要**人的判断**。推导引擎负责排除错误答案和记录诚实过程，不负责替人做审美选择。**审美是约束挤压后自然剩余的东西，不是引擎计算的结果。**
