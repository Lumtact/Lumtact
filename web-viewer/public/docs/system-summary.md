# 水之波光 · 触境 · 体系总纲
## The Water's Gleam · Tactile Realm · System Summary v10.0.0
**声明**：本体系致力于建立一套不可伪造的推导诚实。美是约束挤压后的自然剩余，诚实是挤压过程的不可妥协。

---

## 体系关系图

```
哲学（公理系统）
    ↓ 提供第一原理与推导纪律
约束库（知识库）
    ↓ 提供硬边界与量化阈值
推导引擎（方法论）
    ↓ 提供从目的到形态的推导协议
实例档案（应用记录）
    ↑ 验证/质疑/反馈 → 触发约束库/引擎迭代
    ↑ 不可反向修改哲学或约束库
```

**流向规则**：
- **知识从上往下流**：公理决定约束，约束决定推导，推导决定实例。
- **质疑从下往上流**：实例发现问题 → 质疑约束或引擎 → 通过版本迭代修正。
- **实例永远不能反向修改公理。** 若实例与公理冲突，要么是实例推导错误，要么是公理需要升级——升级通过新版本实现，不原地修改。

## 防撕裂机制
为了防止“设计规范是一回事，代码写出来是另一回事”的撕裂，本体系强制执行以下机制：
### 1. 来源标注强制化
任何设计值若无 `[PHYS]/[ENG]/[GENE]/[PURPOSE]` 标注，视为无效。这防止了"仓库里的值找不到白皮书支撑"的撕裂。
### 2. 实例档案分离
具体值只存在于 `archive.md`，不出现在哲学或约束文档中。这防止了"白皮书被具体值污染"的撕裂。
### 3. 审计清单十一审
每个实例档案必须通过十一审，其中**目的适配审**和**来源标注审**是元审——它们检查"推导过程是否诚实"，不检查"结果是否好看"。
### 4. 版本锚点冻结
每部文档冻结后不再原地修改。修正通过新版本实现，旧版本保留为历史。这防止了"哲学偷偷变化以适应实现"的撕裂。
### 5. 质疑单向流
实例可以质疑约束（"这个约束在此场景下似乎不适用"），但质疑必须通过版本迭代解决，不能临时打补丁。这防止了"实现倒逼哲学妥协"的撕裂。

## 概念生命周期

```
概念诞生 → 进入约束库/引擎 → 实例验证 → 
    ├── 验证通过 → 保留，成为稳定约束
    ├── 验证失败 → 标记为 [DEPRECATED]，下一版本移除
    └── 被新约束替代 → 彻底删除，不残留为幽灵
```

**删除纪律**：
- 被淘汰的概念从全文档彻底移除。
- 实例档案中保留历史记录，但标注 `[OBSOLETE: 被X替代]`。
- 不允许"这个值我们不用了，但代码里还留着"的幽灵状态。

## 版本绑定规则

实例档案必须声明其依赖的体系版本。

```yaml
实例档案绑定声明:
  哲学: @ v10.0.0
  约束库: @ v10.0.0
  推导引擎: @ v10.0.0
```

**兼容性规则**：
- 约束库升级（如新增约束）→ 所有依赖的实例档案需复审。
- 推导引擎升级（如优化流程）→ 实例档案可选择性复用新流程，旧记录保留。
- 哲学升级（公理变更）→ 全体系重构，所有实例档案需重新推导。
